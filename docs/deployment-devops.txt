DEPLOYMENT:
il codice verra' interamente containerizzato in due container separati tra fe e be, questo permette di aggiornare i due pezzi separatamente, 
specialmente il webserver di prod, separandolo dai framework usati nel be e rendendo il patching ed il mantenimento piu' facili da gestire.
Inoltre la separazione permetterebbe di gestire loadbalancers, eventuali improvement con sessioni condivise (es. redis) ed altre implementazioni 
future in maniera piu' facile.
poiche' dobbiamo anche mantenere un release cycle veloce e vogliamo evitare di creare problemi con ambienti diversi (versioni di python, librerie, 
os sul quale si sviluppa != os sul server, etc.) utilizziamo lo stesso pattern anche nel development utilizzando i devcontainers

GITFLOW:
- i developers si occuperanno di riportare le modifiche in maniera appropriata attraverso merge, rebase, cherrypick, etc. tra i diversi branch.
- i developers useranno dei branch feature/<name> facendo un branch a partire da quello develop per sviluppare/testare le funzionalita'
- quando una feature e' completata il branch feature/<name> verra' mergiato verso develop
- quando il branch develop viene mergiato verso il branch preprod il codice verra' automaticamente buildato (voce del verbo buildare) 
  e deployato (idem) sul server come da automazioni CI/CD
- quando il branch develop viene mergiato su main il codice e' accettato per PROD
- quando dal branch PROD vengono fatte tagged release viene assegnato alle release un numero di versione (semantic versioning) e il codice viene deployato sul server
- il pattern di aggiornamento dell'applicazione e' un rebuild e restart, upgrade in place

BRANCHES:
- main: tracking del codice principale
- develop: e' il branch dal quale i developers partiranno per creare i branch feature/<name> e sul quale andranno risolti i merge-conflicts
- preprod: e' il branch sul quale viene mergiato develop e dal quale facciamo i deployment in PREPROD
- tagged release da main: qui gestiamo il versioning e i deployment di produzione.
